# CLAUDE.md

このプロジェクトで作業するときは、以下のルールを必ず守ってください。

前提資料: `docs/current-state-audit.md`(現状棚卸し)と`docs/superpowers/specs/2026-07-16-rewrite-architecture-design.md`(新アーキテクチャ設計、いずれもDiscord_Management_Bot_V2リポジトリ)。実装方針で迷ったらまずこれらを参照する。

## 禁止事項

- 依頼されていないファイルを変更しない。修正が必要だと思ったら、先に提案して許可を得ること。
- ライブラリや依存パッケージを勝手に追加しない。追加が必要な場合は、理由を説明して確認を取ること。
- ハードコードやマジックナンバーは使用しないこと。
- エラーは必ず握り潰しを行わないようにすること。
- Dashboard UIでユーザー・チャンネル・話者等のIDを直接テキスト入力させないこと。必ずAPI経由で一覧取得したセレクター経由で選択させる(設計書§5)。
- 能力ビットフラグ(`packages/shared/src/capabilities.ts`)の既存ビット位置を再利用・並べ替えしないこと。追加は末尾への追記のみ(設計書§6.1)。

## 作業の進め方

- 大きな変更(複数ファイルにまたがる変更、設計の変更など)の前には、必ず内容を説明して確認を取ってから実行する。
- 作業は小さく分けて、少しずつ進める。一度に大量の変更をしない。
- コードを変更したら、必ずテストを実行して結果を報告する。
- テストは単体テストだけではなく、ユースケースを考えた複合テストも使用すること。
- Dashboard側のUIを変更した場合は、手動で画面を動作確認すること。

## Git運用ルール

- 作業を始める前に必ず新しいブランチを切る。mainブランチで直接作業しない。
- 一機能で一コミットをし、なるべく細かく履歴を残すこと。
- 全ての変更(バグ修正・機能追加問わず)は必ずissueを発行してから作業を開始すること。
- マージは通常のmerge commitで統一する。squash mergeは使わない(issue単位の実装過程をmainの履歴に残すため)。
- マージ済みブランチは削除しない(履歴・比較・参照用として保持する)。
- PRを作成するときは、リポジトリにあるPRテンプレート(`.github/PULL_REQUEST_TEMPLATE.md`)に従って記述する。
- Issueを作成するときは、リポジトリにあるIssueテンプレート(`.github/ISSUE_TEMPLATE/`)に従って記述する。
- 原則、git worktree は使用しないこと(通常のブランチ切り替えで対応する)。
- Drizzleのマイグレーションファイルは必ず `pnpm db:generate` で生成し、ファイル名を手打ちしないこと。

## コーディング規約

- 既存コードのスタイル(命名規則・インデント・書き方)に合わせる。
- TypeScript strict モード必須。全パッケージESM(`"type": "module"`)。
- Discord IDは常に`text`型、タイムスタンプは常に`timestamptz`。Postgresの`enum`型は使わず、`text` + `CHECK`制約で表現する(`docs/current-state-audit.md`§6のDB運用ルールを踏襲)。
- パッケージ間依存の方向を守る: `router/` → `application/` → `domain/`、`discord/` → `application/` → `domain/`。基盤パッケージ(`core`/`db`/`shared`/`config`/`dashboard-access`)はドメインパッケージに依存しない(設計書§3)。

## 返答スタイル

- 説明は短く簡潔に。結論を先に述べる。
- 長い前置きや不要な補足は省略する。
